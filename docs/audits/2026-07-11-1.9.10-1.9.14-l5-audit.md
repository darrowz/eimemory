# eimemory 1.9.10-1.9.14 L5 Code Audit

## Scope

Reviewed commits `c92f989` through `6686419`, covering autonomous code-patch
evidence, external-knowledge safety, OpenClaw completion gates, version
alignment, measured autonomous-learning closure, atomic governance state, and
safe web transport.

## Version Findings

- 1.9.10 correctly added executed code-patch preflight, subject-state binding,
  deployment health, and rollback evidence.
- 1.9.11 correctly added knowledge source/safety gates before capability use.
- 1.9.12 added the OpenClaw completion gate, but `eimemory/version.py` was not
  bumped in that commit. 1.9.13 restored file alignment; 1.9.14 is aligned.
- 1.9.14 correctly changed missing capability replay executors to `not_run` and
  removed hard-coded autonomous-learning pass scores.

## Closure Defects Found

1. L5 readiness counted failed and `not_run` replay records as replay volume.
2. L5 readiness counted any assessment artifact, even incomplete or untrusted.
3. Closure rehearsal returned success and wrote task/skill evidence when weak
   replay never executed.
4. Patch metrics counted generic promotions and trusted status-only deployment.
5. Rollback metrics trusted status-only records instead of executed ledgers.
6. Outcome attribution accepted a successful trace whose verifier failed.
7. L5 assessment accepted a boolean-only replay claim.
8. Production Runtime had no capability replay executor, making CLI replay
   permanently `not_run` outside tests.
9. Replay executors could claim pass with contradictory hit/observation data or
   without a traceable evidence source.
10. Stable replay loop IDs reused the first persisted result forever. A later
    failure, `not_run`, or recovered pass could not replace the decision input.

## Resolution

Releases 1.9.15 and 1.9.16 fail closed at each decision boundary, add a production
case-specific executor backed by verified outcome traces, separates rehearsal
success from production task success, require trusted assessment, patch, and
rollback evidence, and persist replay results per execution batch. Readiness
uses the latest result for each case. These changes preserve autonomous
evolution behavior; they only remove evidence shortcuts used by stage and
metric decisions.
