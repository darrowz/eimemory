# eimemory 0.11.0 Full Autonomous Learning System Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` for parallel task execution, or `superpowers:executing-plans` for inline execution. Track steps with checkbox syntax and keep tests green after every phase.

**Goal:** Upgrade eimemory/OpenClaw from outcome-driven rule evolution into a bounded, proactive, continuous autonomous learning system that observes signals, creates its own learning goals, researches with evidence, runs sandbox experiments, evaluates candidates, promotes safe improvements, tracks capability curves, and rolls back regressions.

**Architecture:** Reuse the existing 0.10.0 outcome evolution, rule evolution, code sandbox, evaluation, scheduler, and OpenClaw hook layers. Add a durable learning-loop state machine around them: `observe -> prioritize -> self-model -> goals -> research -> sandbox -> eval -> promotion -> ledger -> regression-watch -> retention`. LLMs may synthesize and rank, but system-owned evidence, replay, gates, and ledgers decide promotion.

**Tech Stack:** Python 3.13, existing eimemory `Runtime`, `RuntimeStore`, SQLite/JSONL storage abstractions, pytest, existing OpenClaw bridge/RPC, existing immutable honxin deployment flow.

## Relationship to Existing Versions

- 0.10.0 already provides outcome traces, diagnosis, replayable candidates, safe action gates, code sandbox, OpenClaw outcome hooks, and nightly outcome summaries.
- The earlier 0.10.1 plan is valuable as the local learning kernel: self-model, curiosity, research, sandbox experiment, eval, and distillation.
- This 0.11.0 plan supersedes 0.10.1 as the complete implementation plan. It incorporates the 0.10.1 kernel, fixes its repository paths, and adds the missing production-grade pieces: loop state, idempotency, evidence trust, data retention, OpenClaw policy injection, demotion, scheduler safety, and longitudinal capability curves.

## What "Autonomous" Means

This version should make the system genuinely self-driving in the bounded engineering sense:

- It can start learning cycles without a direct user correction.
- It can discover repeated weaknesses from local outcomes, recall misses, test failures, watcher signals, and configured public sources.
- It can create learning goals, gather evidence, run offline experiments, score candidates, and promote L0/L1/L2 improvements without waiting for manual approval when the configured machine gates pass.
- It cannot automatically perform L3 actions: external messages, spending, account/auth changes, private data export, device actions, or irreversible deletion.
- Production deployment, live service restart, and system prompt/prompt-policy mutation are L2 fully authorized gated rollout actions: they may be generated, shadow-tested, canaried, applied, monitored, and rolled back by the system after strict evidence, regression, health, and blast-radius gates pass. Human review is not required for L2.

If all acceptance tests pass and scheduler jobs are enabled, the system should continuously improve memory policy, tool-routing policy, eval coverage, local playbooks, source policies, prompt policy, code/default behavior, and production deployment state. It still produces audit briefs for L2 changes, but those briefs are records of autonomous action rather than approval blockers.

## Ten-Pass Design Review

1. **Autonomy trigger:** The loop must run from scheduler/watchers, not only after user feedback.
2. **State machine:** Every cycle must be resumable and idempotent by `loop_id`.
3. **Evidence trust:** LLM synthesis is never evidence; claims need source records, evals, files, commits, URLs, or traces.
4. **Promotion separation:** Candidate generation, evaluation, and promotion must be separate modules.
5. **Regression/anti-learning:** Promoted improvements can expire, demote, disable, or create rollback requests.
6. **OpenClaw integration:** Learned policy must affect task behavior through `searchPolicy`/policy suggestions before generic recall.
7. **Data growth control:** Signals, research notes, candidates, and watcher artifacts need dedupe, caps, TTL, and compaction.
8. **Safety tiers:** L0/L1 may apply locally; L2 is fully authorized after gated rollout checks pass; L3 blocks and asks for approval.
9. **Offline-first:** The full loop must pass without network, paid APIs, or a real LLM.
10. **Measurable improvement:** Capability ledger must show before/after scores, trend, evidence count, and regression count.

## Runtime Invariants

- Every learning run has one durable `learning_loop` record.
- Every mutating step is idempotent by `(loop_id, step_name, semantic_key)`.
- Only one active learning loop may run per scope unless explicitly forced.
- A candidate cannot evaluate itself.
- A candidate cannot promote without an evaluation record.
- A promoted record must include rollback metadata.
- LLM output must be schema-validated and labeled as synthesis.
- Network collectors are disabled by default and must support dry-run.
- Scheduler jobs must support dry-run, max-write limits, and timeout.
- Learned policies enter OpenClaw through policy suggestions, not raw bulk memory injection.
- No watcher or collector may store unsanitized secrets, credentials, raw private transcripts, or raw media.

## Authority Model

- `L0`: Record-only learning artifacts, score updates, replay cases, local reports.
- `L1`: Low-risk local assets such as eval fixtures, disabled source configs, SOP drafts, skill drafts, memory rules, and tool-route drafts.
- `L2`: Fully authorized autonomous action after machine gates pass. Includes code changes, scheduler changes, default behavior changes, OpenClaw routing changes, enabled source policies, production deployment/restart, and system prompt or prompt-policy mutation. Must use gated rollout: evidence bundle, eval report, canary/shadow result, rollback plan, health check, and audit record. Human approval is not required.
- `L3`: External sends/posts/emails/comments, spending, auth/token/OAuth changes, private data export, device actions, irreversible deletion, or privilege expansion. Always blocked pending explicit approval.

## L2 Gated Rollout Requirements

Any L2 deployment, restart, code change, scheduler enablement, or system prompt/prompt-policy mutation must pass all gates:

- Evidence: at least one `T0` or `T1` source, or two independent `T2/T3` sources.
- Evaluation: target capability improves or expected failure mode is reduced.
- Safety: safety score `>= 0.95`.
- Regression: regression score `>= 0.95`.
- Blast radius: canary scope or single-service release first; no broad multi-service rollout in one step.
- Rollback: executable rollback command or disable flag is stored before apply.
- Health: post-apply health check and smoke test must pass.
- Timeout: rollout stops automatically on timeout or unavailable health endpoint.
- Audit: create `promotion_request`, `learning_eval`, and `capability_score` records with applied artifact ids.
- Cooldown: repeated failed rollout for the same semantic key opens a cooldown and blocks reapply.

L2 authorization policy:

- Passing all gates means the system is authorized to apply immediately.
- Codex briefs, patch bundles, and review notes are generated for audit, rollback, and later human understanding, not as approval gates.
- If a gate fails, the system must not ask for approval to bypass it; it records a blocked candidate and either retries after cooldown or generates a lower-risk experiment.
- If post-apply health or attributed outcomes regress, rollback is automatic when an executable rollback exists.
- If rollback fails, the system opens an incident and blocks further candidates with the same semantic key.

L2 prompt changes must also include:

- Prompt diff with before/after text.
- A shadow prompt eval on replay cases.
- A prompt injection safety check.
- A rollback pointer to the previous prompt version.
- Policy attribution so future bad outcomes can identify the prompt version.

## Evidence Trust Tiers

- `T0 first_party_outcome`: outcome traces, user corrections, verified replay results.
- `T1 local_eval`: unit tests, benchmark results, CLI smoke, production smoke.
- `T2 local_artifact`: repo files, commits, logs, config, local memory records.
- `T3 official_source`: official docs, official release notes, standards.
- `T4 research_source`: papers, benchmark reports, reputable technical posts.
- `T5 community_source`: GitHub discussions, forums, blogs.
- `T6 llm_synthesis`: synthesis only; cannot be promotion evidence alone.

## Final Judgment

If this plan is implemented and the Done Criteria pass, eimemory/OpenClaw will have a real bounded autonomous learning loop: it can proactively notice gaps, decide what to learn, gather evidence, test changes, apply low-risk improvements, autonomously run gated L2 prompt/code/deployment rollouts, and monitor whether changes helped. It will not ask for human approval for L2, because the gates, canary, rollback, audit trail, and regression watch are the approval mechanism. L3 remains blocked because those actions create external, financial, credential, device, or irreversible consequences outside the learning system's rollback envelope.
